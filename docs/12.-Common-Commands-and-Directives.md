# 1. SQLcl Aliases

Several of the most commonly used commands below are configured as SQLcl Aliases and saved in the file `scripts/xml/sqlcl-project-aliases.xml` in this repo. To load these aliases in your SQLcl environment, run the SQLcl `alias load` command as follows:

## 1.1.  Load Aliases

```bash
alias load scripts/xml/sqlcl-project-aliases.xml
```

## 1.2.  Save Aliases

```bash
alias save scripts/xml/sqlcl-project-aliases.xml
```

## 1.3.  List Aliases

You can check if they were loaded successfully by running `alias list` command and see if you have aliases starting with `prj_`

```
SQL> alias list
prj_install
prj_mr
prj_status
prj_sync
SQL> 
```

# 2. prj_* Alias Reference

All custom aliases defined in `scripts/xml/sqlcl-project-aliases.xml` use the `prj_` prefix. They serve as productivity shortcuts and workarounds for known SQLcl Project and Liquibase limitations. Load them with `alias load scripts/xml/sqlcl-project-aliases.xml` (see [Section 1.1](#11--load-aliases)).

## 2.1. prj_compile

**Purpose**: Compiles APEXlang source files into the target APEX application by running `apex import`. This is the primary alias used in AI-assisted APEXlang workflows — edit `.apx` files, compile, fix errors, repeat.

**Parameter**: `app_id` — the APEX application ID.

The alias auto-resolves the source path and workspace name from `apex_applications`, so you only need to provide the app ID.

Example:
```sql
conn -name proj_dev
prj_compile 110
```

See also: [6.8. apex import](#68-apex-import)

## 2.2. prj_drift_cleanup

**Purpose**: Runs `scripts/bash/tools/drift_cleanup.sh` — a batch script that automates several drift-detection cleanup steps (removing ORDS noise, whitespace changes, and other export artefacts). Less frequently needed than the other aliases, but essential during drift analysis.

Example:
```sql
prj_drift_cleanup
```

See also: [06. Mitigating Production Drift](06.-Mitigating-Production-Drift.md)

## 2.3. prj_exp_app

**Purpose**: Smart APEX export wrapper. Runs `project export -o APEX.<app_id>` and then applies a version-specific workaround:
- **SQLcl 26.1.2**: re-exports the app using `apex export -exptype APEXLANG` to work around a known bug where `project export` corrupts binary files and leaves stale files from previous exports.
- **Other versions**: runs `apex validate` on the exported APEXlang.

**Parameter**: `app_id` — the APEX application ID.

Example:
```sql
conn -name proj_dev
prj_exp_app 110
```

> **Note**: `prj_exp_app_old` is a deprecated predecessor of this alias that manually deleted the export folder before re-exporting (using `rm -fR` / PowerShell `Remove-Item`). It has been superseded by `prj_exp_app`.

See also: [6.6. apex export](#66-apex-export)

## 2.4. prj_force_apex

**Purpose**: Forces re-deployment of all APEX applications on the next `prj_install`, even if Liquibase considers them already up to date. Does this by deleting the relevant APEX changeset records from `databasechangelog` and `databasechangelog_actions`. Prompts for confirmation before proceeding.

After running, either:
- Run `prj_install` to immediately re-install all APEX apps, or
- Run `prj_sync` to re-mark them as current without re-deploying.

Example:
```sql
conn -name proj_dev
prj_force_apex
```

## 2.5. prj_install

**Purpose**: One-command deployment shortcut. Changes directory to `dist` and runs `@install.sql` — the standard SQLcl Project installer script. This is the 99% use case for non-production deployments.

Example:
```sql
conn -name proj_vm
prj_install
```

See also: [6.5. @install](#65-install)

## 2.6. prj_mr

**Purpose**: Wrapper for `lb mark-next-changeset-ran`. Marks the next pending Liquibase changeset as executed without actually running it. The result is always the same: the changeset is skipped on the next `prj_install` run. Common use cases:

- A change was already applied manually outside the deployment pipeline
- A change is already present in the database as a result of drift
- A changeset is not applicable in the current environment (e.g., a VM used only for syntax validation where SSO or environment-specific configuration changes serve no purpose)

Example:
```sql
conn -name proj_test
prj_mr
```

See also: [4.5. lb mark-next-changeset-ran(-sql)](#45-lb-mark-next-changeset-ran-sql)

## 2.7. prj_rm_logs

**Purpose**: Deletes all `*.log` files found anywhere in the repository. Liquibase sometimes writes log files into the `dist` folder that should be git-ignored but occasionally end up tracked. This alias removes them cleanly.

Example:
```sql
prj_rm_logs
```

## 2.8. prj_rm_ords

**Purpose**: Removes spuriously staged ORDS changes from the working tree and index using `git restore`. SQLcl Project 26.1 has a known bug where it stages ORDS schema files even when there are no actual ORDS changes. Run this alias to clean up the noise before committing.

Example:
```sql
prj_rm_ords
```

## 2.9. prj_status

**Purpose**: Wrapper for `lb status`. Shows all pending (not yet applied) Liquibase changesets — effectively a preview of what `prj_install` would deploy today.

Example:
```sql
conn -name proj_test
prj_status
```

See also: [4.2. lb status](#42-lb-status)

## 2.10. prj_sync

**Purpose**: Wrapper for `lb changelog-sync`. Marks **all** changesets in the changelog as executed without applying them. Used when baselining a new environment or handling drift scenarios.

Example:
```sql
conn -name proj_prod
prj_sync
```

See also: [4.4. lb changelog-sync](#44-lb-changelog-sync)

## 2.11. prj_validate

**Purpose**: Validates APEXlang syntax without importing into the database. Wraps `apex validate`. Useful in AI-assisted workflows to confirm `.apx` file changes are syntactically correct before committing to a full `prj_compile`.

**Parameter**: `app_id` — the APEX application ID.

Example:
```sql
conn -name proj_dev
prj_validate 110
```

See also: [6.7. apex validate](#67-apex-validate)

# 3. Project Commands

## 3.1. Additional Notes (Project Commands)

- Always run `project export` before `project stage`.
- Review staged files carefully before committing.
- `project gen-artifact` only includes staged files.


## 3.2. project init

Initializes a new SQLcl Project structure. Used only once per repository.

Example:
```bash
project init -name real_sqlcl -schemas cla_apex,cla_utilities
```


## 3.3. project export

Exports database objects or APEX applications from the connected environment into the repository.

conn -name proj_dev
project export -o APEX.110

## 3.4. project export -list

Shows what objects export command would exported, but does not run export

Example:
```bash
conn -name proj_dev
project export -list
```


## 3.5. project stage

Compares the repo state with the database and prepares staged files for a release.

Example:
```bash
conn -name proj_dev
project stage
```


## 3.6. proj stage add-custom

The most common custom changes:

- DML on “fixed” tables, for example, insert 15 invoice statuses into INVOICE_STATUS table
- Handle objects that SQLcl project does not handle natively, for example:
- Run `Alter System`
- Create/Drop APEX Workspace
- Create/Drop Scheduled Jobs, AQ Tables, Refresh Materialized Views
- Stop/Start Jobs and APEX Automations
- ...and much more…

```bash
conn -name proj_dev
proj stage add-custom -file-name  test.sql
```
> After running add-custom command above, open the file dist/releases/next/changes/<your_branch>/_custom/test.sql for edit and add your code after LB header.


## 3.7. project release

Creates a release directory for grouping staged changes into a deployable unit.

Example:
```bash
project release -name release_25_2
```


## 3.8. project gen-artifact

Generates the deployable ZIP artifact from the release folder.

Example:
```bash
project gen-artifact -release release_25_2
```


## 3.9. project deploy

Deploys the ZIP artifact to the connected environment. Should be executed only with the correct DBA-level deployer account.

Example:
```bash
conn -name proj_test
project deploy -file dist/releases/release_25_2.zip -verbose
```


# 4. Liquibase Commands

## 4.1. Additional Notes (Liquibase)

- Always connect then `cd dist` before running LB.
- Use `-changelog-file releases/main.changelog.xml` except for drop-all.


## 4.2. lb status

Shows which Liquibase changesets are pending or already applied.

Example:
```bash
conn -name proj_test
cd dist
lb status -changelog-file releases/main.changelog.xml
```
Alias: `prj_status`

## 4.3. lb update

Applies pending changesets to the current environment.

Example:
```bash
conn -name proj_test
cd dist
lb update -changelog-file releases/main.changelog.xml
```


> This is a command that runs from dist/instal.sql . Unlikely you will need to run it directly

## 4.4. lb changelog-sync

- Marks all existing changesets as executed **without applying them**. Mostly used for baseline creation.
- `changelog-sync-sql` shows what changesets will be applied 

Example:
```bash
conn -name proj_prod
cd dist
lb changelog-sync -changelog-file releases/main.changelog.xml
```
Alias: `prj_sync`


## 4.5. lb mark-next-changeset-ran(-sql)

Marks the next unexecuted changeset as executed. This is mostly used in the following scenario:
- You ran deployment and it failed, let's say on a unique constaraint
- You ran DML and get rid of the issues
- Now you can run  `lb mark-next-changeset-ran` to mark the failed changeset as executed
- `mark-next-changeset-ran(-sql)` shows which changeset will be run, but does not run it

Example:
```bash
conn -name proj_test
cd dist
lb mark-next-changeset-ran -changelog-file releases/main.changelog.xml
```

Alias: `prj_mr`

## 4.6. lb drop-all

Drops all objects in the connected schema.  
- Usually runs in  a VM or disposable environment.
- Also used in **Hotfix** flow to prepare DEV environment for applying a hotfix 

Example:
```bash
conn -name proj_vm
cd dist
lb drop-all
```

# 5. Git Commands

## 5.1. Additional Notes (Git)
- Tag prod releases for rollbacks.
- Commit staged changes after review.
- You can run git commands from inside SQLcl using `!git ...`


## 5.2. git checkout

Switches the working directory to the given branch.

Example:
```bash
git checkout feature_1
```


## 5.3. git checkout -b

Creates a new branch and switches into it.

Example:
```bash
git checkout feature_1
```


## 5.4. git add

Stages modified files for commit.

Example:
```bash
git add .
```


## 5.5. git commit

Commits staged files with a message.

Example:
```bash
git commit -m "Added missing constraint to SCOTT.EMP"
```


## 5.6. git push

Sends local commits to the remote repository.

```bash
#Example ( first push):
git push --set-upstream origin fix_missing_grants
#Subseqent pushes 
git push
```


## 5.7. git tag

Creates an annotated tag – usually for PROD releases.

Example:
```bash
git tag -a v25.2 -m "Production baseline after April release"
git push origin v25.2
```


## 5.8. git restore

Restores files from another branch or commit.

Examples:

```bash
git restore --source=main --worktree --staged -- dist/releases/ords
git restore --source=main --worktree --staged dist/releases/
git restore --staged dist/releases/
git clean -fd dist/releases/
```


## 5.9. git merge

Merges another branch into the current one.

Example 1: Merge feature to main - Usually should be done on remote server via PR/MR

```bash
git checkout main
git merge feature_1
```

Example 2 : Merge main into feature - Usually should be done to resolve merge conflicts 

```bash
git feature_1
git merge main
```

# 8. Environment Variables

## 8.1. Use \${VAR_NAME} syntax

``` sql
begin
   cla_apex.cla_configuration_pkg.upsert (
        p_instance_id =>1 ,
        p_key_type_id => 1,
        p_key         => 'ams_server',
        p_description => 'Local AMS (APEX Messaging Service) server URL',
        p_value       => 'https://${AMS_SERVER_FQDN}',
        p_key_group   => 'ams'
    );  
end;
/
```


SQLcl Project substitutes **OS-level environment variables** during deployment.  
These variables **must be defined in the operating system**, not inside SQLcl or SQL scripts.

If a variable is **missing**, SQLcl Project **does not fail** — instead, it inserts the literal placeholder:

```
'${AMS_SERVER_FQDN}'
```

This can silently corrupt configuration values across DEV/TEST/PROD.

---

## 8.2. Setting Environment Variables

### 8.2.1. Unix / Linux / macOS
```bash
export AMS_SERVER_FQDN='127.0.0.1'
export API_KEY='abc123'
```

### 8.2.2. Windows PowerShell
```powershell
$env:AMS_SERVER_FQDN = "127.0.0.1"
$env:API_KEY = "abc123"
```

### 8.2.3. Windows CMD
```cmd
set AMS_SERVER_FQDN=127.0.0.1
set API_KEY=abc123
```

These must be set **on the machine that runs the deployment**.

---

## 8.3. ⚠ Important Warning

SQLcl Project **will NOT throw an error** if a required variable is missing.

It simply writes the literal text `${VAR_NAME}` into the database.

---

## 8.4. Validating Environment Variables in Changesets (Best Practice)

Use Liquibase preconditions to ensure variables are set:

```sql
--preconditions onFail:CONTINUE
--precondition-sql-check expectedResult:0     SELECT count(*)    FROM dual    WHERE '${AMS_SERVER_FQDN}' = '$'||'{'||'AMS_SERVER_FQDN'||'}'
```

**Behavior:**
- Missing variable → condition TRUE → **changeset skipped**  
- Valid variable → condition FALSE → **changeset runs**

---
